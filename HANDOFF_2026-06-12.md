## 2. What changed today (by file)

### Purchase engine
- `good360_roster/good360_autobuy_v2.py`
  - **`_live_purchase_approval_blocker()` — hard approval gate.** Every live purchase
    re-verifies at the point of purchase that the dashboard shows the customer
    `status='active' AND in_rotation=1`. Fails closed (any verification error =
    refusal) and Telegrams the operator if it ever trips. NEVER weaken or bypass.
    Context: a roster-sync bug let the engine attempt a purchase for "Mike's skins
    charities", whom the operator had removed from rotation.
  - **`_run_checkout_via_daemon()` — new default engine.** POSTs an org_config to the
    daemon's `/test_checkout` (deterministic Playwright executor in
    `good360_daemon.py::_checkout_inner`, screenshots every step, evidence-based
    success detection). `AUTOBUY_ENGINE` env/setting selects the engine; default is
    `daemon`. The LLM agent requires explicit `AUTOBUY_ENGINE=devtools_agent`.
  - **Fake-success guard.** An agent SUCCESS without a confirmation number is mapped
    to `failed_checkout` ("UNVERIFIED SUCCESS REJECTED"). Reason: the gpt-5.1 agent
    reported a completed $5,976.16 purchase (event #53) for an order that NEVER
    existed — verified by logging into the customer's Good360 Order History. The
    false records were corrected (purchase_attempts #32, truck_events #53 → missed)
    and the customer's undeserved 7-day cooldown removed. **The customer (Hope 4
    Humanity / Berneitha) received a false "Purchase Confirmed" email — the operator
    knows; make sure it was communicated.**
  - `OrgContext.billing_address` field added (QuickBeed billing dict now flows to the
    daemon engine, which matches saved billing addresses by zip).

- `good360_devtools_agent.py` (LLM path — now opt-in only)
  - Prompt now carries the empirically-derived site playbook (see §6), explicit
    authorization framing, a worked numeric budget comparison (the model once refused
    claiming $5,976 > $6,200), retry-before-declaring-stuck rules, and a hard rule
    that SUCCESS requires the verbatim order number.
  - History: this agent produced FIVE distinct fabrications in one night (fake
    "human-only step" refusal, fake budget math, fake success, "test card data"
    rationalization, single-snapshot stuck conclusions). Treat it as untrusted.

### Mission Control (`missioncontrol/`)
- **`customer_readiness.py` (new).** Validates full QuickBeed records against
  everything checkout consumes: credentials (presence + email shape after trim),
  cards (number presence, **Luhn checksum + 13–19 digit length**, expiry parseable +
  unexpired, CVV, name, full billing address), warehouse address, phone, budget,
  truck selection, checkout answers. Distinguishes blockers vs warnings. Persists
  `data_ok` / `data_issues` / `cards_meta` / `data_checked_at` on `customers`;
  Telegrams the operator on NEW blockers (deduped); daily sweep + revalidation on
  every `fetch_full()` and within ~5 min of a record changing (poll loop hook in
  `server_v2.py`, changed-id tracking in `quickbeed.py::incremental_sync`).
- **`admin_routes.py`**
  - `expiry_mmyy` normalizer (in customer_readiness, single source of truth): handles
    the packed `exp_month=1030, exp_year=None` shape ("10/30" collapsed into one
    field). This bug blocked ALL purchases 5/18 → 6/11 ("no usable payment_methods").
  - Good360 login email `.strip()` at the org-config + test-purchase boundaries
    (trailing space in a stored username made Good360's form reject it as invalid).
  - New endpoints: `GET/POST /api/admin/data-issues[/check]` (flag list + manual
    sweep), `GET /customers/<id>/card-details` (FULL card reveal, super-admin,
    double-audited like /credentials), `POST /customers/<id>/revalidate` (resync one
    customer from QuickBeed + payment-readiness verdict),
    `DELETE /customers/<id>/cooldown` (manual cooldown graduation — clears BOTH
    stores, see §6), and the rotation PATCH now pushes `sync_to_roster()` immediately.
- **`quickbeed_roster_sync.py`** — roster eligibility (`auto_buy_global`) now requires
  `status='active' AND in_rotation=1`. Previously QuickBeed-active alone made a
  customer purchasable regardless of the operator's toggle (the Mike's-skins
  incident).
- **`db.py`** — migrations for `data_ok`, `data_issues`, `data_checked_at`,
  `cards_meta` (non-sensitive card summary only; PAN/CVV are never stored locally).
- **UI (`static/admin.js/.html/.css`)** — Card column (last4/network/⚠) + "⚠ data"
  badge in the customers list; Payment methods panel: "Resync & validate" +
  "Reveal card details" buttons with verdict banner; "✕ clear" cooldown buttons on
  the roster Cool-off chips and customer detail; fixed the spinning-wheel cursor on
  disabled autobuy toggles (`cursor: not-allowed`; busy cursor only during in-flight
  saves via a `saving` class).

### Monitor (`good360_monitor.py`)

### Configuration

## 5. Technical debt / lower priority

- **Two-queue architecture.** The monitor's `find_next_queued_customer` (dashboard)
  picks a customer and logs `QUEUE ASSIGN`, but `roster_orchestrator.handle_truck_event`
  independently picks its own org from roster.db — the QUEUE ASSIGN log is effectively
  advisory. Eligibility is now consistent (both honor in_rotation), but ordering can
  differ. Unify these (likely: make the roster the single queue, delete the monitor's).
- **Re-arm friction.** Failed attempts consume the availability event (alerted list)
  and set in-flight cooldowns, so retrying a continuously-available truck is manual
  (skill §2). Consider auto-rearm on hard failures with a retry budget.
- **Readiness flags don't gate roster eligibility.** A `data_ok=0` customer can still
  be picked (the processor decline is the backstop). Consider
  `auto_buy_global = active AND in_rotation AND data_ok != 0`.
- **Infra (from the morning's diagnosis, still open):**
  - intake service: compose publishes `5000:5000` but the Flask app listens on 8080
    → externally unreachable; healthcheck fails on a Playwright/Chromium SIGTRAP
    (container is 4 weeks old; recreate or switch to an HTTP healthcheck).
  - stale `app-*` compose project: `app-intake-1` still running a duplicate intake
    since May 10 + 4 Created containers → `docker compose -p app down`.
  - daemon port 5002 is published to the internet and being probed by scanners —
    bind to 127.0.0.1 or put behind Caddy.
  - monitor memory ~2.3 GB (long-lived browser; watch it).
- The daemon SUCCESS path puts the page-evidence string into `confirmation_number`
  when no order number is parsed — consider making the daemon extract the order
  number explicitly and/or post-purchase verification against Order History.

## 6. Operational knowledge you MUST have

- **Single-file bind mounts pin the inode.** `good360_monitor.py`,
  `good360_roster/good360_autobuy_v2.py`, `good360_devtools_agent.py` etc. are
  mounted as single files into the monitor — after editing them,
  `docker compose up -d --force-recreate monitor` (a plain restart is NOT enough;
  verify with `md5sum` host vs container). Directory mounts (`./missioncontrol`)
  only need a service restart (gunicorn reload); static JS/HTML/CSS serve live.
- **`settings_bootstrap.py` overrides env from the settings table** at startup
  (QUICKBEED_DRY_RUN, AUTOBUY_ENGINE, DEVTOOLS_*…). Check BOTH places.
- **Two cooldown stores:** dashboard `customers.cooldown_until` AND roster
  `nonprofits.status='cooldown'/cooldown_until`. The UI "✕ clear" button and the
  DELETE /cooldown endpoint clear both; manual SQL must too.
- **Site quirks (catalog.good360.org)** — full list in the memory/skill and the
  devtools agent prompt: React-controlled inputs (use prototype value setter +
  events), react-select everywhere (no native <select>), 3-step single-page wizard
  whose Continue buttons silently no-op on unregistered state, login URL STAYS on
  /sign-in after success (check password-field-gone, not URL), first Add-to-Cart
  pops a "Truck Quote Request" modal, no /cart page (mini-cart aside only).
- **Proof-of-purchase standard:** a success claim without an order number / page
  evidence is a failure. When in doubt, log into the customer account (read-only,
  creds via missioncontrol org-config API with an audit `reason`) and check
  `/marketplace/my-account/orders`.
- **Standing rules:** purchases only for dashboard-approved customers (active +
  in_rotation; gate fails closed); all form/payment data comes from the customer's
  QuickBeed record (normalization like trimming/whitespace is OK, inventing values
  is not); never use a real card for tests.
